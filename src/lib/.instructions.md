# Core Library Instructions

## `docker-parser.ts` — The Command Parser

This is the heart of the simulation. All state mutations flow through `parseCommand()`.

### Adding a New Docker Command

1. Add a new `case` in the `switch` block inside `parseCommand()`
2. Create a `handleXxx(parts, state, updateState)` function following existing patterns
3. Update `getHelpText()` to document the new command
4. Return a `CommandResult` with appropriate `success`, `output`, and optional `error`
5. If the command changes state, call `updateState()` with the new containers/images

### Important Constraints

- Never allow direct state mutation outside the parser
- Validate all user input; handle edge cases like missing args, duplicate names, port conflicts
- Keep command output formatted for monospace terminal display

## `types.ts` — Domain Types

All domain types are defined here. Key types:

- `DockerContainer` — id, name, image, status (`running | stopped | exited`), ports, created, command
- `DockerImage` — id, name, tag, size, created, layers
- `TerminalLine` — id, type (`command | output | error | success`), content, timestamp
- `CommandResult` — success, output, error
- `Tutorial` / `TutorialStep` — guided learning configuration
- `Achievement` — gamification reward definitions

## `tutorials.ts` — Tutorial Definitions

Each tutorial has an array of `TutorialStep` objects with:
- `expectedCommand`: string or string[] of acceptable commands
- `validation`: optional function to check state after the command
- `hints`: progressive hints shown to the user

## `achievements.ts` — Achievement Definitions

Achievements are unlocked by a `requirement` function that checks `AchievementCheckData`.
Achievement IDs must match tutorial IDs when a tutorial completion triggers an achievement.
